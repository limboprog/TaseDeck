# 📘 Проект «Agent Control Plane» — Спецификация платформы

## 1. 🎯 Задача платформы

**Обеспечить полную наблюдаемость, контроль и оптимизацию работы AI-агентов через персонализированную память решений и динамическую фильтрацию инструментов.**

Платформа решает три фундаментальные проблемы современной экосистемы MCP/агентов:

| Проблема | Решение |
|----------|---------|
| **Tool overload** (200+ тулов → LLM тонет в метаданных) | Динамический ранжировщик отдаёт агенту только топ-5 релевантных тулов |
| **Black box decisions** (почему агент удалил файл — неизвестно) | Детерминированный decision logger с обязательным полем `reason` |
| **Отсутствие обратной связи** (какие тулы работают лучше — никто не знает) | Статистика эффективности комбинаций токенов/шагов/успеха |

## 2. 📋 Функциональные требования (по слоям)

### Слой 1. Персональный реестр агентов и MCP-инструментов

**Задача:** Хранить в едином месте всё, что относится к работе конкретного пользователя с агентами.

| Требование | Детали | Приоритет |
|------------|--------|-----------|
| **1.1** | Реестр **пользовательских сессий** с агентами (по проектам/workspace) | P0 |
| **1.2** | Реестр **MCP-серверов**, доступных пользователю (имя, версия, команда запуска, credentials reference) | P0 |
| **1.3** | Реестр **скиллов** (skills: название, описание, версия, dependencies) | P0 |
| **1.4** | Реестр **вызовов инструментов** (какой тул, когда, кем, с какими параметрами, какой результат) | P0 |
| **1.5** | Реестр **цепочек вызовов** (последовательность тулов для решения задачи) | P1 |
| **1.6** | Хранение **мета-информации** (cost в токенах, latency, success/failure) для каждого вызова | P0 |
| **1.7** | Версионирование конфигураций агентов и MCP-серверов (change tracking) | P2 |

### Слой 2. Decision Logger (логи решений с explainability)

**Задача:** Записывать не только факт вызова инструмента, но и **почему** агент его выбрал.

| Требование | Детали | Приоритет |
|------------|--------|-----------|
| **2.1** | При вызове любого MCP-тула агент **обязан** предоставить поле `reason` (естественный язык, 50-200 символов) | P0 |
| **2.2** | Платформа проверяет наличие `reason`, при отсутствии — reject вызова с ошибкой | P1 |
| **2.3** | Сохраняется полная трассировка принятия решения: запрос пользователя → рассуждения агента → выбранный тул → результат | P0 |
| **2.4** | Поддержка **вложенных решений** (sub-decisions): если агент планирует последовательность действий, каждое действие логируется отдельно | P1 |
| **2.5** | Decision logger должен быть **агностичен к провайдеру** (работает с Claude Code, Cursor, OpenCode, любым MCP-клиентом) | P0 |
| **2.6** | API для **ретроспективного анализа** сессии: "покажи все решения, где тул был rejected" | P1 |

### Слой 3. Динамический ранжировщик инструментов (Dynamic Tool Selector)

**Задача:** По текущему запросу пользователя + истории его решений + анализу кода проекта вернуть **топ-5 наиболее релевантных MCP-тулов**, которые агент должен увидеть.

| Требование | Детали | Приоритет |
|------------|--------|-----------|
| **3.1** | Перехват запроса пользователя к агенту перед отправкой в LLM | P0 |
| **3.2** | Извлечение **контекста проекта**: язык, фреймворки, структура папок, ключевые эндпоинты | P0 |
| **3.3** | Поиск в **истории пользователя** (слой 1) похожих запросов и успешных комбинаций тулов | P0 |
| **3.4** | Ранжирование доступных MCP-тулов по формуле: `score = w1*query_similarity + w2*history_boost + w3*project_relevance + w4*success_rate` | P0 |
| **3.5** | Возврат ранжированного списка (топ-5) **вместо** полного списка всех тулов агента | P0 |
| **3.6** | Фильтрация **опасных тулов** для данного контекста (например, не показывать `delete_branch`, если пользователь не в `admin` role) | P1 |
| **3.7** | Поддержка **negative boosting** — если тул никогда не использовался для этого типа задач, его ранг падает | P2 |
| **3.8** | A/B-тестирование разных ранжировок на пользовательских сессиях | P2 |

### Слой 4. Статистика эффективности (Analytics)

**Задача:** Дать пользователю и администратору данные о том, какие инструменты и комбинации работают лучше, а какие — хуже.

| Требование | Детали | Приоритет |
|------------|--------|-----------|
| **4.1** | Дашборд с метриками для каждого MCP-тула: количество вызовов, success rate, средняя latency, средние токены | P0 |
| **4.2** | Анализ **комбинаций тулов** (биграммы/триграммы): какие два/три тула чаще всего вызываются последовательно | P0 |
| **4.3** | Сравнение эффективности: для запросов типа X комбинация (grep + call_graph) решает задачу в среднем за 2 шага, а (semantic_search + ripgrep) — за 5 шагов | P0 |
| **4.4** | **Cost analysis** по проектам/пользователям: сколько токенов сэкономил dynamic selector по сравнению с full tool list (projected savings) | P1 |
| **4.5** | **Heatmap** использования инструментов по времени суток / по типу задачи | P2 |
| **4.6** | Экспорт статистики в CSV / JSON для интеграции с внешними системами | P1 |

### Слой 5. UI (Веб + Десктоп)

**Задача:** Предоставить пользователю единый интерфейс для управления своей платформой, просмотра истории, настройки правил.

| Требование | Детали | Приоритет |
|------------|--------|-----------|
| **5.1** | **Дашборд сессий**: список всех сессий работы с агентами с фильтром по дате, проекту, исходу (успех/ошибка) | P0 |
| **5.2** | **Просмотр сессии в timeline**: визуализация цепочки "запрос → решение → тул → результат → следующий запрос" | P0 |
| **5.3** | **Детальный просмотр решения**: текст `reason` + параметры вызова + результат + затраченные токены | P0 |
| **5.4** | **Управление правилами фильтрации**: пользователь может manually pin/unpin тулы для определённых проектов | P1 |
| **5.5** | **Настройка ранжировщика**: веса w1/w2/w3/w4 могут быть изменены пользователем через слайдеры | P2 |
| **5.6** | **Дашборд статистики**: визуализация метрик из слоя 4 (графики, heatmap, cost savings) | P0 |
| **5.7** | **Интеграция с IDE**: плагин для VS Code / Cursor, показывающий "рекомендованные тулы для этого запроса" прямо в процессе | P1 |
| **5.8** | **Desktop-клиент** (Electron/Tauri) для локального запуска платформы без облака | P1 |

### Слой 6. Интеграция и API

**Задача:** Платформа должна легко встраиваться в существующий workflow разработчика.

| Требование | Детали | Приоритет |
|------------|--------|-----------|
| **6.1** | **MCP-прокси** — платформа работает как reverse proxy между агентом и MCP-серверами | P0 |
| **6.2** | **REST API** для всех операций (реестр, логи, статистика, ранжирование) | P0 |
| **6.3** | **Webhook** для событий: `tool_called`, `decision_made`, `anomaly_detected` | P1 |
| **6.4** | **OpenTelemetry экспорт** трасс в Jaeger / Grafana | P2 |
| **6.5** | **CLI** для управления из терминала (инициализация, запуск прокси, просмотр логов) | P1 |
| **6.6** | **Docker-образ** для одноминутного развёртывания | P0 |

### Слой 7. Безопасность и приватность

| Требование | Детали | Приоритет |
|------------|--------|-----------|
| **7.1** | Все данные хранятся **локально** (SQLite по умолчанию, опционально Postgres) | P0 |
| **7.2** | Шифрование credentials для MCP-серверов (API keys, tokens) | P0 |
| **7.3** | Audit log всех действий администратора/пользователя с платформой | P1 |
| **7.4** | Опциональный **cloud-mode** с синхронизацией между устройствами (user-controlled) | P2 |

## 3. 🏗️ Архитектурные принципы

| Принцип | Описание |
|---------|----------|
| **Local-first** | Все данные по умолчанию на машине пользователя. Cloud — опционально |
| **MCP-native** | Платформа сама является MCP-сервером (или прокси) и не требует модификации агентов |
| **Explainability-by-design** | Нет вызова тула без `reason`. Иначе reject. |
| **Pluggable ranker** | Алгоритм ранжирования можно заменить на свой через plugin API |
| **KISS** | SQLite как основное хранилище, минимум внешних зависимостей |

## 4. 🧩 Интерфейсы (кратко)

### 4.1 Web UI (основные экраны)

| Экран | Что показывает |
|-------|----------------|
| **Dashboard** | Сводка: активные сессии, топ-5 тулов по использованию, cost savings за неделю |
| **Sessions** | Список всех сессий с фильтрами. Клик → timeline визуализации |
| **Session detail** | Timeline "запрос → решение → тул → результат" с `reason` на каждом шаге |
| **Registry** | Список MCP-серверов и скиллов пользователя |
| **Analytics** | Графики эффективности тулов, heatmap комбинаций, cost breakdown |
| **Rules** | Настройка ранжировщика (веса, pin/unpin тулов для проектов) |

### 4.2 CLI (минимальный интерфейс)

```bash
# Инициализация
acp init

# Запуск proxy
acp proxy --port 3000

# Просмотр сессий
acp session list --project myproject

# Просмотр конкретной сессии
acp session view session_123 --timeline

# Статистика
acp stats --tool grep --since 7d

# Управление реестром
acp registry add mcp --name github --cmd "npx -y @modelcontextprotocol/server-github"
```

### 4.3 API (основные эндпоинты)

```http
# Реестр
GET    /api/v1/sessions
GET    /api/v1/sessions/{id}/timeline
GET    /api/v1/tools
POST   /api/v1/tools/{id}/pin

# Decision logging
POST   /api/v1/decisions
  Body: { session_id, query, reasoning, tool_name, params, result, tokens, latency }

# Ранжирование
POST   /api/v1/rank
  Body: { query, project_context, history_limit }
  Response: [{ tool_name, score, reason_why_recommended }]

# Статистика
GET    /api/v1/stats/tools?timeframe=7d
GET    /api/v1/stats/combinations?top=10
```

## 5. 📈 Успех измеряется по метрикам

| Метрика | Цель (бенчмарк) | Как измеряется |
|---------|----------------|----------------|
| **Input token reduction** | Снижение на 80% (10K → 2K токенов на запрос) | Сравнение с baseline (full tool list) |
| **Tool selection accuracy** | Топ-1 рекомендуемый тул совпадает с реально использованным в 80% случаев | Precision@1 |
| **Session success rate** | ↑ на 30% (меньше итераций на задачу) | Среднее число вызовов тулов на задачу |
| **Explainability coverage** | 100% вызовов имеют `reason` | Доля вызовов с непустым reason |
| **User retention** | 70% пользователей активны через 30 дней | DAU/MAU |

| Фаза | Срок | Что включает |

## ✅ Критерии готовности MVP

- [ ] Прокси перехватывает запросы к MCP-серверам
- [ ] Каждый вызов тула логируется с полем `reason`
- [ ] Логи хранятся в SQLite
- [ ] CLI показывает список сессий и timeline конкретной сессии
- [ ] Базовый ranker (по query similarity через эмбеддинги) возвращает топ-5 тулов
- [ ] Агент (Claude Code / Cursor) работает через прокси без изменений
